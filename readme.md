# Практическое занятие: Контейнеризация приложений с использованием Docker

## Предварительные требования

Перед началом необходимо убедиться в наличии установленного Docker Desktop. Приложение можно загрузить с официального сайта https://www.docker.com/get-started/. После установки следует запустить Docker Desktop и дождаться полной инициализации сервиса.

Для пользователей из РФ требуется настройка зеркал Docker Hub. Данная настройка выполняется в разделе Settings, далее Docker Engine. В конфигурационный файл необходимо добавить следующий блок:

```json
{
  "registry-mirrors": [
    "https://mirror.gcr.io",
    "https://dockerhub.timeweb.cloud/"
  ]
}
```

После внесения изменений следует нажать кнопку Apply and restart.

### Примечание для пользователей Windows

Все команды, представленные в данном занятии, выполняются в терминале PowerShell или командной строке Windows. Docker Desktop на Windows 10 использует WSL 2 в качестве backend, однако непосредственная работа в WSL-терминале не требуется.

Для перехода между директориями в PowerShell используется стандартная команда cd с указанием полного или относительного пути:

```
cd C:\путь\к\проекту\demo1_simple_python
```

При возникновении проблем с производительностью файловой системы рекомендуется размещать проекты внутри файловой системы WSL.

## Часть 1. Введение в контейнеризацию

### Теоретические основы

Docker представляет собой платформу для разработки, доставки и запуска приложений в изолированных средах, называемых контейнерами. Контейнер содержит приложение и все его зависимости, что обеспечивает идентичное поведение программы независимо от окружения, в котором она выполняется.

Ключевым отличием контейнеризации от виртуализации является отсутствие необходимости эмулировать аппаратное обеспечение и запускать полноценную операционную систему. Контейнеры используют ядро хостовой операционной системы, что делает их значительно легковеснее виртуальных машин.

### Архитектура Docker

Архитектура Docker построена на клиент-серверной модели и включает следующие компоненты:

**Docker Daemon** представляет собой фоновый процесс, управляющий образами, контейнерами, сетями и томами. Именно этот компонент выполняет все операции по созданию и запуску контейнеров.

**Docker Client** является интерфейсом командной строки, через который пользователь взаимодействует с Docker Daemon. Команды, вводимые пользователем, передаются демону для выполнения.

**Docker Image** представляет собой неизменяемый шаблон, содержащий файловую систему и метаданные, необходимые для создания контейнера. Образы состоят из слоев, каждый из которых содержит изменения относительно предыдущего слоя.

**Docker Container** является запущенным экземпляром образа. Контейнер имеет собственную изолированную файловую систему, сетевой стек и пространство процессов.

**Docker Registry** служит хранилищем образов. Наиболее известным публичным реестром является Docker Hub.

### Проверка работоспособности Docker

Для проверки корректности установки Docker следует выполнить следующие команды в терминале:

```
docker --version
```

Данная команда выводит версию установленного Docker.

```
docker info
```

Эта команда отображает подробную информацию о конфигурации Docker, включая количество контейнеров и образов.

## Часть 2. Контейнеризация простого Python-скрипта

### Постановка задачи

Рассмотрим процесс создания Docker-образа на примере простого Python-скрипта. Данный пример позволит понять базовые принципы работы с Dockerfile и основные команды Docker.

### Демонстрация 1: Простой Python-скрипт

Перейдем в директорию с первым примером:

```
cd demo1_simple_python
```

Рассмотрим структуру проекта. В директории находятся три файла:

Файл **main.py** содержит исходный код приложения. В данном случае это простой скрипт, выполняющий сложение двух чисел.

Файл **Dockerfile** содержит инструкции по сборке образа. Рассмотрим его содержимое подробнее:

```dockerfile
FROM python:3.12-slim
LABEL maintainer="student"
WORKDIR /usr/src/app
COPY main.py ./
CMD ["python", "./main.py"]
```

Инструкция FROM указывает базовый образ, на основе которого будет создаваться новый образ. В данном случае используется официальный образ Python версии 3.12 в облегченной версии slim.

Инструкция LABEL добавляет метаданные к образу.

Инструкция WORKDIR устанавливает рабочую директорию внутри контейнера. Все последующие команды будут выполняться относительно этой директории.

Инструкция COPY копирует файлы из контекста сборки в файловую систему контейнера.

Инструкция CMD определяет команду, которая будет выполнена при запуске контейнера.

Файл **.dockerignore** содержит список файлов и директорий, которые не должны включаться в контекст сборки. Это позволяет уменьшить размер образа и ускорить процесс сборки.

### Сборка образа

Для создания образа используется команда docker build:

```
docker build -t python_demo:1.0 .
```

Флаг -t задает имя и тег образа. Точка в конце команды указывает на текущую директорию как контекст сборки.

В процессе сборки Docker выполняет каждую инструкцию из Dockerfile, создавая промежуточные слои. Результатом является готовый образ, который можно увидеть в списке:

```
docker image ls
```

### Запуск контейнера

Для запуска контейнера на основе созданного образа используется команда docker run:

```
docker run python_demo:1.0
```

После выполнения команды в терминале отобразится результат работы скрипта. Контейнер выполнит команду, указанную в инструкции CMD, и завершит работу.

### Просмотр структуры образа в Docker Desktop

В графическом интерфейсе Docker Desktop можно просмотреть детальную информацию об образе, включая его слои, размер и файловую структуру. Для этого следует перейти в раздел Images и выбрать созданный образ.

## Часть 3. Контейнеризация веб-приложения FastAPI

### Особенности контейнеризации веб-приложений

Контейнеризация веб-приложений имеет ряд особенностей по сравнению с консольными программами. Веб-приложение должно принимать входящие соединения, что требует настройки сетевого взаимодействия между контейнером и хостовой системой.

### Демонстрация 2: FastAPI Hello World

Перейдем в директорию со вторым примером:

```
cd demo2_fastapi_hello
```

В данном примере представлено минимальное FastAPI-приложение с двумя эндпоинтами.

Рассмотрим Dockerfile для веб-приложения:

```dockerfile
FROM python:3.12-alpine
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
WORKDIR /usr/src/app
COPY requirements.txt .
RUN pip install --upgrade pip && pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "80"]
```

Обратим внимание на несколько важных аспектов:

Использование образа с тегом alpine означает применение облегченной версии, основанной на дистрибутиве Alpine Linux. Такие образы имеют минимальный размер.

Переменная окружения PYTHONDONTWRITEBYTECODE=1 отключает создание файлов .pyc, что уменьшает размер образа.

Переменная PYTHONUNBUFFERED=1 отключает буферизацию вывода Python, что позволяет видеть логи приложения в реальном времени.

Инструкция RUN выполняет команды в процессе сборки образа. В данном случае происходит установка зависимостей из файла requirements.txt.

Параметр --host 0.0.0.0 в команде запуска uvicorn указывает, что сервер должен принимать соединения на всех сетевых интерфейсах контейнера.

### Сборка и запуск

Выполним сборку образа:

```
docker build -t fastapi_demo:1.0 .
```

Запустим контейнер с проброской порта:

```
docker run -d -p 80:80 --name fastapi_app fastapi_demo:1.0
```

Флаг -d запускает контейнер в фоновом режиме (detached mode).

Флаг -p 80:80 осуществляет проброску порта. Первое число указывает порт на хостовой машине, второе — порт внутри контейнера.

Флаг --name задает имя контейнера для удобства дальнейшего управления.

### Проверка работоспособности

После запуска контейнера приложение доступно по адресу http://127.0.0.1/. Документация API в формате Swagger UI доступна по адресу http://127.0.0.1/docs.

Для просмотра запущенных контейнеров используется команда:

```
docker container ps
```

Для просмотра логов контейнера:

```
docker logs fastapi_app
```

### Остановка и удаление контейнера

Для остановки контейнера:

```
docker stop fastapi_app
```

Для удаления остановленного контейнера:

```
docker rm fastapi_app
```

## Часть 4. Многоконтейнерные приложения с Docker Compose

### Теоретические основы Docker Compose

Реальные приложения редко функционируют изолированно. Как правило, они взаимодействуют с базами данных, кэширующими системами, очередями сообщений и другими сервисами. Docker Compose позволяет описать все компоненты приложения в едином конфигурационном файле и управлять ими как единым целым.

Конфигурация Docker Compose хранится в файле docker-compose.yml в формате YAML. Файл описывает сервисы, сети, тома и другие ресурсы, необходимые для работы приложения.

### Основные команды Docker Compose

Сборка образов для всех сервисов:
```
docker compose build
```

Запуск всех сервисов в фоновом режиме:
```
docker compose up -d
```

Просмотр статуса сервисов:
```
docker compose ps
```

Просмотр логов конкретного сервиса:
```
docker compose logs -f web
```

Выполнение команды внутри контейнера сервиса:
```
docker compose exec web sh
```

Остановка и удаление всех контейнеров:
```
docker compose down
```

Остановка с удалением томов:
```
docker compose down -v
```

### Демонстрация 3: FastAPI с PostgreSQL

Перейдем в директорию с третьим примером:

```
cd demo3_fastapi_postgres
```

Данный пример демонстрирует типичную архитектуру веб-приложения, состоящего из двух сервисов: веб-приложения на FastAPI и базы данных PostgreSQL.

Рассмотрим структуру файла docker-compose.yml:

```yaml
services:
  web:
    build:
      context: .
      dockerfile: Dockerfile
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
    ports:
      - "8000:8000"
    depends_on:
      - db
    environment:
      - DATABASE_URL=postgresql+asyncpg://demo_user:demo_password@db:5432/demo_db

  db:
    image: postgres:16-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      - POSTGRES_USER=demo_user
      - POSTGRES_PASSWORD=demo_password
      - POSTGRES_DB=demo_db
    ports:
      - "5432:5432"

volumes:
  postgres_data:
```

Секция services описывает контейнеры, которые будут созданы. В данном случае определены два сервиса: web и db.

Сервис web собирается из Dockerfile, расположенного в корне проекта. Параметр depends_on указывает, что данный сервис зависит от сервиса db и должен запускаться после него.

Сервис db использует готовый образ postgres:16-alpine из Docker Hub. Параметр volumes монтирует том для сохранения данных базы между перезапусками контейнера.

Секция volumes на верхнем уровне определяет именованные тома. Том postgres_data будет использоваться для хранения данных PostgreSQL.

Обратим внимание на строку подключения к базе данных. Вместо localhost или IP-адреса используется имя сервиса db. Docker Compose автоматически создает сеть, в которой сервисы могут обращаться друг к другу по именам.

### Запуск многоконтейнерного приложения

Выполним сборку и запуск:

```
docker compose up -d --build
```

Флаг --build указывает на необходимость пересборки образов перед запуском.

Проверим статус сервисов:

```
docker compose ps
```

Обе службы должны находиться в состоянии running.

### Проверка работы приложения

Приложение доступно по адресу http://127.0.0.1:8000/. Документация API доступна по адресу http://127.0.0.1:8000/docs.

Эндпоинт /health возвращает статус подключения к базе данных, что позволяет убедиться в корректной работе связки FastAPI и PostgreSQL.

### Работа с базой данных

Для подключения к PostgreSQL внутри контейнера используется команда:

```
docker compose exec db psql -U demo_user -d demo_db
```

После подключения можно выполнять SQL-запросы. Для просмотра списка таблиц:

```
\dt
```

Для выхода из psql:

```
\q
```

### Тестирование API

Через Swagger UI по адресу http://127.0.0.1:8000/docs можно протестировать работу API:

1. Создать товар через эндпоинт POST /items
2. Получить список товаров через GET /items
3. Получить конкретный товар через GET /items/{item_id}

Данные сохраняются в PostgreSQL и остаются доступными после перезапуска контейнеров благодаря использованию тома.

### Просмотр логов

Для просмотра логов веб-приложения в реальном времени:

```
docker compose logs -f web
```

Для просмотра логов базы данных:

```
docker compose logs -f db
```

### Остановка приложения

Для остановки всех сервисов с сохранением данных:

```
docker compose down
```

Для остановки с удалением томов (все данные будут потеряны):

```
docker compose down -v
```

## Часть 5. Дополнительные сведения

### Полезные команды Docker

Просмотр всех контейнеров, включая остановленные:
```
docker container ps -a
```

Удаление всех остановленных контейнеров:
```
docker container prune
```

Удаление неиспользуемых образов:
```
docker image prune
```

Удаление всех неиспользуемых ресурсов:
```
docker system prune
```

Просмотр использования дискового пространства:
```
docker system df
```

### Вход в работающий контейнер

Для получения доступа к командной строке внутри контейнера:

```
docker exec -it <container_name> sh
```

Флаг -it обеспечивает интерактивный режим работы с терминалом.

### Переменные окружения и безопасность

В рассмотренных примерах учетные данные базы данных указаны непосредственно в конфигурационных файлах. В производственной среде следует использовать механизмы управления секретами, такие как Docker Secrets или переменные окружения, передаваемые при запуске контейнера.
